Overview of the Windows Port

HOPR is a Fortran/C application originally developed and tested exclusively on Linux. This document records every source-level change required to compile and run hopr.exe under Windows using the MSYS2 UCRT64 environment (GFortran 14, GCC 14, Ninja, CMake 4.x).

Porting strategy — minimal invasive. All Windows-specific code is isolated behind #ifdef _WIN32 / IF(WIN32) guards. Linux behaviour is entirely unchanged. No existing logic was removed or restructured.
Upstream documentation note. The upstream docs/documentation/userguide/installation.md still states “HOPR supports Linux-based systems only”. This Windows port is an unofficial extension and is not reflected in the official documentation.

The changes fall into eight distinct areas:

  1. Runtime stack management — src/stacksize.c (new Windows no-op path)
  2. Fortran/C interface — src/globals.f90 (BIND(C) declaration)
  3. Compiler flags — cmake/SetCompiler.cmake (disable -fstack-arrays on Windows)
  4. Library detection & build commands — cmake/SetLibraries.cmake
  5. CGNS library search paths — cmake/FindCGNS.cmake
  6. Build integration & linker flags — src/CMakeLists.txt + CMake cache
  7. Periodic mesh connectivity — src/mesh/mesh_connect.f90 (ASSOCIATED() fallbacks for GFortran 15 Windows)
  8. Node initialisation — src/mesh/mesh_vars.f90 (zero-init Node%tmp to prevent false BC assignments from uninitialised heap memory)

Runtime Stack Management

src/stacksize.c
Modified Added Windows no-op path for setstacksizeunlimited()

On Linux, HOPR calls setstacksizeunlimited() at startup to raise the process stack to its hard limit via POSIX setrlimit(). This call is essential because GFortran compiled with -fstack-arrays places large temporary arrays on the stack.

On Windows, there is no POSIX setrlimit / RLIMIT_STACK. The stack size is instead fixed at link time in the PE (Portable Executable) header. Attempting to include <sys/resource.h> on MinGW/UCRT64 will fail with a compilation error. The function therefore becomes a no-op; the required stack size is set via the linker flag -Wl,--stack,268435456 (256 MB) instead.

src/stacksize.c full file — 17 lines 
#ifdef _WIN32
/* On Windows the stack size is managed by the PE header / linker flags.   */
/* There is no POSIX setrlimit, so this is a no-op.                        */
void setstacksizeunlimited(void) { }
#else
#include <sys/resource.h>

void setstacksizeunlimited(void)
{
   struct rlimit limit;
   getrlimit(RLIMIT_STACK, &limit);
   limit.rlim_cur=limit.rlim_max;
   setrlimit(RLIMIT_STACK, &limit);
}
#endif
Key point: The #ifdef _WIN32 guard is defined automatically by MinGW-w64 / UCRT64 on all Windows builds. No CMake flag or manual define is required.

Fortran/C Interface Declaration

src/globals.f90
Added C-interoperable interface block for setstacksizeunlimited (lines 103–106)

Fortran cannot directly call a C function without an explicit interface declaration that uses BIND(C). This interface block was added to globals.f90 so that the Fortran entry point hopr.f90 can call the C function. The declaration itself is platform-neutral; the platform-specific behaviour lives entirely in the C translation unit.

src/globals.f90 — lines 103–106 added block 
INTERFACE
  SUBROUTINE setstacksizeunlimited() BIND(C)
  END SUBROUTINE setstacksizeunlimited
END INTERFACE

Fortran Compiler Flags — Disable -fstack-arrays on Windows

cmake/SetCompiler.cmake
CMake Conditional -fstack-arrays flag — Windows branch (lines 82–93)

GFortran’s -fstack-arrays flag instructs the compiler to allocate all local and temporary arrays on the thread stack rather than the heap. This is a significant performance optimisation on Linux where setstacksizeunlimited() has already expanded the stack to its hard limit.

On Windows the stack size is fixed at link time (256 MB, see Section 6). Enabling -fstack-arrays with this constraint causes stack overflow when processing large meshes. The flag is therefore omitted from all release-type build configurations on Windows.

cmake/SetCompiler.cmake — lines 82–93 (GNU compiler block) WIN32 branch added 
  ! initialize all variables as signalling NaNs to force the user to correctly initialize
  ! On Windows the stack size cannot be raised at runtime (setstacksizeunlimited is a no-op).
  ! -fstack-arrays puts all temporary/allocatable arrays on the stack, which causes stack overflow
  ! for large meshes on Windows. Use heap allocation there instead.
  IF(WIN32)
    SET(CMAKE_Fortran_FLAGS_RELEASE        "${CMAKE_Fortran_FLAGS}     -O3 ${HOPR_INSTRUCTION} -finline-functions")
    SET(CMAKE_Fortran_FLAGS_RELWITHDEBINFO "${CMAKE_Fortran_FLAGS} -g  -O3 ${HOPR_INSTRUCTION} -finline-functions -ffpe-trap=invalid,zero,overflow -fbacktrace")
    SET(CMAKE_Fortran_FLAGS_PROFILE        "${CMAKE_Fortran_FLAGS} -pg -O3 ${HOPR_INSTRUCTION} -finline-functions")
  ELSE()
    SET(CMAKE_Fortran_FLAGS_RELEASE        "${CMAKE_Fortran_FLAGS}     -O3 ${HOPR_INSTRUCTION} -finline-functions -fstack-arrays")
    SET(CMAKE_Fortran_FLAGS_RELWITHDEBINFO "${CMAKE_Fortran_FLAGS} -g  -O3 ${HOPR_INSTRUCTION} -finline-functions -fstack-arrays -ffpe-trap=invalid,zero,overflow -fbacktrace")
    SET(CMAKE_Fortran_FLAGS_PROFILE        "${CMAKE_Fortran_FLAGS} -pg -O3 ${HOPR_INSTRUCTION} -finline-functions -fstack-arrays")
  ENDIF()
Build type Linux flag set includes Windows flag set includes
Release -O3 -finline-functions -fstack-arrays -O3 -finline-functions
RelWithDebInfo -g -O3 -finline-functions -fstack-arrays -ffpe-trap=... -fbacktrace -g -O3 -finline-functions -ffpe-trap=... -fbacktrace
Profile -pg -O3 -finline-functions -fstack-arrays -pg -O3 -finline-functions
Debug / Sanitize Unchanged — same flags on both platforms

Library Detection & External Build Commands

cmake/SetLibraries.cmake
CMake Skip readelf / nm ZLIB check on Windows (line 121)

After finding a system HDF5 library, the build system normally inspects the HDF5 binary with readelf (Linux) or nm (macOS) to determine whether HDF5 was compiled with ZLIB support and thus whether ZLIB must be linked separately. Neither tool is available in a standard Windows environment. A NOT WIN32 guard skips the inspection entirely; on Windows ZLIB detection falls back to a hard-coded result of not found (i.e., ZLIB is not linked separately).

cmake/SetLibraries.cmake — HDF5 ZLIB detection block (approx. lines 121–129) NOT WIN32 guard added 
  IF(NOT "${HDF5_C_LIBRARY_hdf5_c}" STREQUAL "")
  IF(NOT "${HDF5_C_LIBRARY_hdf5_c}" STREQUAL "" AND NOT WIN32)
    IF(APPLE)
      EXECUTE_PROCESS(COMMAND nm -gU      ${HDF5_C_LIBRARY_hdf5_c}
                      COMMAND grep inflate ...)
    ELSE()
      EXECUTE_PROCESS(COMMAND readelf -Ws ${HDF5_C_LIBRARY_hdf5_c}
                      COMMAND grep inflate ...)
    ENDIF()
  ELSE()
    SET(GREP_RESULT 1)   ! treat as “not linked against zlib”
  ENDIF()
CMake Platform-conditional build command for self-built HDF5 (approx. lines 196–200)

When HOPR falls back to compiling HDF5 from source (because no system HDF5 is found), the upstream code used make -jN as the build command. GNU make is not available on Windows; the cross-platform equivalent is cmake --build.

cmake/SetLibraries.cmake — HDF5 self-build command WIN32 branch added 
  IF(WIN32)
    SET(HDF5_BUILD_CMD ${CMAKE_COMMAND} --build <BINARY_DIR> -j${N})
  ELSE()
    SET(HDF5_BUILD_CMD make -j${N})
  ENDIF()
CMake Platform-conditional build command for self-built LAPACK (approx. lines 349–353)

Identical change to the LAPACK self-build path for the same reason: make is replaced by cmake --build when building on Windows.

cmake/SetLibraries.cmake — LAPACK self-build command WIN32 branch added 
  IF(WIN32)
    SET(LAPACK_BUILD_CMD ${CMAKE_COMMAND} --build <BINARY_DIR> -j${N})
  ELSE()
    SET(LAPACK_BUILD_CMD make -j${N})
  ENDIF()
In the current working build HDF5 and LAPACK/OpenBLAS are both satisfied by the MSYS2 UCRT64 package manager (pacman), so these self-build paths are not exercised. The guards remain important for reproducibility on machines without pre-installed packages.

CGNS Library Search Paths

cmake/FindCGNS.cmake
Added MSYS2 UCRT64 and MinGW64 paths to find_path / find_library

The upstream FindCGNS.cmake searched only standard Unix paths (/usr/local/include, /usr/include, etc.). These do not exist as Windows filesystem paths. Two MSYS2 environments are added so that CMake can locate the CGNS library installed via pacman without requiring the user to set environment variables manually.

cmake/FindCGNS.cmake — full file (include search) lines 8–17 
find_path(CGNS_INCLUDE_DIR
  NAMES cgnslib.h
  HINTS $ENV{CGNS_DIR}/include
        $ENV{CGNS_ROOT}/include
        /usr/local/include
        /usr/include
        C:/msys64/ucrt64/include
        C:/msys64/mingw64/include
  DOC "CGNS include directory"
)
cmake/FindCGNS.cmake — library search lines 20–30 
find_library(CGNS_LIBRARY
  NAMES cgns
  HINTS $ENV{CGNS_DIR}/lib
        $ENV{CGNS_ROOT}/lib
        /usr/local/lib
        /usr/lib
        C:/msys64/ucrt64/lib
        C:/msys64/mingw64/lib
  DOC "CGNS library"
)
Installation prerequisite: CGNS must be installed in MSYS2 before configuring:
pacman -S mingw-w64-ucrt-x86_64-cgns
If the variable CGNS_DIR or CGNS_ROOT is set in the environment, those paths are probed first and take precedence over the hardcoded MSYS2 paths.

Windows Stack Size — Linker Flag

CMakeCache.txt (build_windows)
Config CMAKE_EXE_LINKER_FLAGS set to -Wl,--stack,268435456

Because setstacksizeunlimited() is a no-op on Windows, the required stack space must be reserved at link time. The flag -Wl,--stack,268435456 passes the --stack 268435456 option to the MinGW linker, which embeds a 256 MB stack reservation in the PE header of hopr.exe.

This value was determined empirically to be sufficient for typical mesh sizes processed without the -fstack-arrays flag active. Users processing extremely large meshes may need to increase this value by reconfiguring with a larger number.

build_windows/CMakeCache.txt — linker flags entry 
CMAKE_EXE_LINKER_FLAGS:STRING=-Wl,--stack,268435456
How to change the stack size: Pass -DCMAKE_EXE_LINKER_FLAGS="-Wl,--stack,<bytes>" when running cmake, or edit the value directly in CMakeCache.txt and re-run ninja.
Example for 512 MB: -Wl,--stack,536870912

CGNS Fortran Module ABI Mismatch — Self-Build Required

cmake/SetLibraries.cmake • CMakePresets.json
Bug MSYS2 libcgns.dll.a has no Fortran module symbols → link failure

When LIBS_USE_CGNS=ON, the build links against C:/msys64/ucrt64/lib/libcgns.dll.a (installed via pacman -S mingw-w64-ucrt-x86_64-cgns). The MSYS2 CGNS package is compiled as a C-only library and does not include the Fortran cgns.mod module file or the corresponding Fortran module symbol table (__cgns_MOD_cg_*).

HOPR's src/readin/readin_CGNS.f90 uses the Fortran module interface (USE CGNS) rather than raw C bindings, so the linker fails with dozens of undefined reference to __cgns_MOD_cg_close_f errors.

Linker error (excerpt)
ld.exe: undefined reference to `__cgns_MOD_cg_version_f'
ld.exe: undefined reference to `__cgns_MOD_cg_open_f'
ld.exe: undefined reference to `__cgns_MOD_cg_close_f' (×40+)
Fix Use self-built CGNS with -DCGNS_ENABLE_FORTRAN=ON

The CMake build system already contains a complete self-build path for CGNS (triggered by LIBS_BUILD_CGNS=ON) that downloads CGNS 4.4.0 from GitHub and compiles it with -DCGNS_ENABLE_FORTRAN=ON. This produces a static libcgns.a and a cgns.mod module file compiled by the same gfortran version used for HOPR itself — eliminating the ABI mismatch.

cmake/SetLibraries.cmake — self-build CGNS arguments (existing, no change needed)
CMAKE_ARGS  -DCMAKE_C_COMPILER=${CMAKE_C_COMPILER}
            -DCMAKE_Fortran_COMPILER=${CMAKE_Fortran_COMPILER}
            -DCGNS_ENABLE_FORTRAN=ON          ! produces cgns.mod compatible with host gfortran
            -DCGNS_BUILD_SHARED=OFF
            -DCGNS_ENABLE_HDF5=ON
            ...
First-time build takes ~5 minutes to download and compile CGNS 4.4.0. Subsequent builds are instant (cmake detects the pre-built libcgns.a). Requires internet access during the initial cmake configure step.
Configure with self-built CGNS:
cmake --preset windows-ucrt64   (uses the new preset added in Section 10)
or manually:
cmake -S . -B build_windows -DLIBS_USE_CGNS=ON -DLIBS_BUILD_CGNS=ON -DCMAKE_EXE_LINKER_FLAGS="-Wl,--stack,268435456"

ASSOCIATED() False Negatives — OrientedNode Pointer Check

src/mesh/mesh_connect.f90
Bug Two-argument ASSOCIATED(P,Q) returns .FALSE. for same-target pointers on Windows

After converting tetrahedral elements to hexahedra via SplitToHex=T and building periodic boundary meshes, ConnectMesh() crashes with:

Runtime abort
Problem with OrientedNode on periodic side (master)!

Root cause: The periodic node orientation logic calls ASSOCIATED(nSide%Node(1)%np, Side%connection%Node(iNode)%np) to check whether the slave node and the master's dummy connection node are the same tNode object after GlobalUniqueNodes pointer remapping. GFortran 15 on Windows returns .FALSE. for this two-argument form in certain pointer-of-struct aliasing patterns even when both arguments target the same memory address. (The one-argument ASSOCIATED(P) null check works correctly; only the identity comparison is affected.)

Affected mesh types: Cartesian meshes using elemtype=104 (linear tetrahedra) with SplitToHex=T and at least one periodic BC.

Fix Three-level fallback chain after the ASSOCIATED() check

Three additional checks are inserted after the ASSOCIATED test in both the slave-side branch and the master-side branch of ConnectMesh(). They fire only when ASSOCIATED returns false; in the common case (Linux or a correct Windows pointer comparison) they are never reached.

  1. %tmp NodeID comparison — after the first pass of ConnectMesh resets all node %tmp to 0 and the second pass assigns a unique integer NodeID per live tNode object, two pointers that target the same object share the same NodeID.
  2. %ind comparison — after GlobalUniqueNodes both merged nodes have ind = MAX(original_inds), which is the same positive value.
  3. SAMEPOINT() coordinate comparison — last resort; uses SpaceQuandt * PP_MeshTolerance tolerance.
src/mesh/mesh_connect.f90 — master-side branch (representative excerpt) 
    ELSE !Side is master periodic, nSide is slave
      fNode=0
      DO iNode=1,nSide%nNodes
        IF(ASSOCIATED(nSide%Node(1)%np,Side%connection%Node(iNode)%np)) fNode=iNode
      END DO
      ! Windows fallback 1: ConnectMesh NodeID — same %tmp = same tNode object
      IF(fNode.EQ.0) THEN
        DO iNode=1,Side%nNodes
          IF(nSide%Node(1)%np%tmp .EQ. Side%connection%Node(iNode)%np%tmp) THEN
            fNode=iNode; EXIT
          END IF
        END DO
      END IF
      ! Windows fallback 2: ind comparison after GlobalUniqueNodes merge
      IF(fNode.EQ.0) THEN
        DO iNode=1,Side%nNodes
          IF(nSide%Node(1)%np%ind .EQ. Side%connection%Node(iNode)%np%ind) THEN
            fNode=iNode; EXIT
          END IF
        END DO
      END IF
      ! Windows fallback 3: coordinate comparison within mesh tolerance
      IF(fNode.EQ.0) THEN
        DO iNode=1,Side%nNodes
          IF(SAMEPOINT(nSide%Node(1)%np%x,Side%connection%Node(iNode)%np%x)) THEN
            fNode=iNode; EXIT
          END IF
        END DO
      END IF
      IF(fNode.EQ.0)CALL abort(__STAMP__,'Problem with OrientedNode on periodic side (master)!')
Test cases fixed by this change (combined with Section 10): CHE_drift_diffusion_explicit-FV/periodic_box_free-stream_split2hex, WEK_PIC_maxwell/2D_periodic_CVWM_split2hex.

Uninitialized Node%tmp — False Periodic-BC Assignments

src/mesh/mesh_vars.f90
Bug Windows heap does not zero-fill; garbage %tmp causes wrong BCs on interior faces

meshTemplate=3 creates six face-centre nodes (9–14) for each Cartesian hex cell via getNewNode(). After allocation, cartmesh.f90 sets node(11)%np%tmp = 20 (eta-minus marker) only when the cell is on the y = 0 boundary. For interior cells the field is left at its post-ALLOCATE value.

On Linux the allocator (glibc) zero-fills new pages, so %tmp = 0 by accident. On Windows (MinGW/CRT) the heap does not zero-fill, so %tmp can contain arbitrary bit patterns. The boundary-condition check in cartmesh.f90 (lines 711–734) reads the decimal digits of %tmp as a bitmask to decide which boundary a side face is on:

cartmesh.f90 — BC bitmask check (simplified)
! iSide=2 checks "tens digit of tmp == 2"  (eta-minus / y=0 flag)
IF( (MOD(node%tmp, 100) / 10) .EQ. iSide ) onBnd = .TRUE.

When an interior face-centre node has a garbage %tmp whose tens digit happens to be 2, the face is falsely tagged as a y-periodic boundary face. SplitTetraToHex then copies the y-periodic BC onto the corresponding sub-hex face, creating a spurious periodic master side with nodes that are entirely interior to the domain. This interior master side gets a dummy connection at y + VVy > 1, outside the domain — no slave node can match it. The sorted SideConnect array therefore contains two identical master sides (from two hexes sharing boundary face nodes) that get incorrectly paired with each other instead of with the true slave, causing the OrientedNode (master) abort.

Fix Initialise Node%tmp = 0 in getNewNode()
src/mesh/mesh_vars.f90 — getNewNode() (1 line added)
ALLOCATE(Node)
Node%ind=0
Node%tmp=0
Node%refCount=0
Why only %tmp? The other integer fields (%ind, %refCount) are already explicitly zeroed in the same subroutine. %tmp was the sole field left uninitialised; it is reused for several different purposes (BC detection bitmask, SFC sort index, ConnectMesh NodeID) and each use assumes a well-defined starting value.
Test cases fixed (combined with Section 9): CHE_drift_diffusion_explicit-FV/periodic_box_free-stream_split2hex, WEK_PIC_maxwell/2D_periodic_CVWM_split2hex.

Windows CMake Presets

CMakePresets.json
Added Two Windows configure presets: full CGNS and no-CGNS

The upstream CMakePresets.json contained only generic debug/release presets with CGNS disabled. Two Windows-specific presets were added:

  • windows-ucrt64 — Release build with self-built CGNS (full functionality). Use this for production hopr.exe.
  • windows-ucrt64-nocgns — Release without CGNS (fast rebuild, ~1 min). Suitable when only Cartesian and Gambit meshes are needed.

Build Instructions (MSYS2 UCRT64)

Prerequisites: MSYS2 UCRT64 shell with the following packages installed:
pacman -S mingw-w64-ucrt-x86_64-toolchain mingw-w64-ucrt-x86_64-cmake mingw-w64-ucrt-x86_64-ninja mingw-w64-ucrt-x86_64-hdf5 mingw-w64-ucrt-x86_64-openblas
Note: Do NOT install mingw-w64-ucrt-x86_64-cgns — use the self-built CGNS instead.
Full build with CGNS support (recommended)
MSYS2 UCRT64 terminal
cd /c/Data/PRJ/hopr-win/hopr-master

# Configure (downloads and builds CGNS ~5 min first time)
cmake --preset windows-ucrt64

# Build
ninja -C build_windows

# Binary location
# build_windows/bin/hopr.exe
Fast build without CGNS (for Cartesian / Gambit meshes only)
MSYS2 UCRT64 terminal
cmake --preset windows-ucrt64-nocgns
ninja -C build_windows_nocgns
Rebuild after source changes only (no reconfigure needed)
MSYS2 UCRT64 terminal
ninja -C build_windows
Known remaining limitations (10 test cases still failing out of 340):
  • 4→1 mortar (3 cases) — NonconformConnectMesh() only supports 2→1 non-conforming interfaces. Test cases: NIG_PIC_maxwell_RK4/PML-2elem-Mortar, NIG_PIC_maxwell_RK4_p_adaption/5elem-with-Mortar-1-to-4, NIG_PIC_maxwell_RK4_p_adaption/PML-5elem-with-Mortar-1-to-4.
  • External mesh files missing (2 cases) — ANSA/Star-CD (Mode=4) and Gambit (Mode=2) test cases that require tildbox.inp / export.neu not bundled in the test suite: NIG_tracking_DSMC/ANSA_box, CHE_DSMC/Rotational_Reference_Frame_Temperature.
  • CGNS input files missing (2 cases) — Mode=3 test cases require CGNS source files not bundled in the test suite: NIG_tracking_DSMC/curved_planar, NIG_tracking_DSMC/mortar_hexpress.
  • nocrosscombination with 3 DEFVAR variables (3 cases) — The DEFVAR substitution engine does not correctly separate combinations when nocrosscombination links three variables simultaneously; the resulting merged mesh has non-conforming inter-zone interfaces. Workaround: split into separate single-variant ini files. Affected: NIG_convtest_poisson/Dielectric_slab_FPC, …/Dielectric_slab_FPC_via_VDL, …/Dielectric_slab_FPC_via_VDL_different_factors.
